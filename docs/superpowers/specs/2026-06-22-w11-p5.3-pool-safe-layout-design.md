# W11 P5.3 — Multi-pool generation + pool-safe auto-layout (design)

**Status:** approved-approach, spec under review
**Date:** 2026-06-22

## Problem

When a user asks the AI to generate a multi-party collaboration ("two pools:
Bank and Customer, with message flows"), Claril produces a **single process**
and folds the parties into lanes — it "skips pools." This is not a model glitch.

Root cause, confirmed:

- `packages/ai-advisor/src/generate-bpmn.ts:10` **explicitly forbids**
  `<collaboration>`, `<participant>`, and pools, and instructs the model to put
  every actor into a `<laneSet>`/`<lane>` of one `<process>`.
- That instruction exists because the downstream layout engine,
  `bpmn-auto-layout@1.3.0` (latest), **cannot** lay out collaborations. Its
  README limitations:
  > - Given a collaboration **only the first participant's process will be laid out**
  > - Message flows are **not laid out**

So a genuine two-pool diagram with message flows currently has no way to be
laid out and would render as disconnected, pool-less boxes — hence the guard.

## Goal

Generate **true multi-pool collaborations** (stacked horizontal pools + dashed
message flows) that render correctly in bpmn-js, matching the standard bpmn.io
layout (reference: `~/Desktop/twopool.png`). Single-party processes keep
today's exact behavior (one process, optional lanes) — we do **not** over-pool.

## Non-goals

- No change to the conversational **editing** planner (it already supports
  pools/message flows via `addPool`/`connect flow:"message"`).
- No per-pool collapse/expand, no swimlane auto-balancing, no message-flow
  label auto-placement beyond a centered label.
- No diagram-interchange editing UI work.

## Design

Two parts: relax the generator, and add a collaboration-aware layout pass that
wraps (not replaces) `bpmn-auto-layout`.

### Part A — Generator (ai-advisor)

Relax `GENERATE_BPMN_SYSTEM_PROMPT` in `generate-bpmn.ts`:

- **Default stays single-process.** Distinct roles/teams/systems **inside one
  organization** are still lanes of one process.
- **Emit a `<collaboration>` with multiple `<participant>`s (each its own
  `<process>`) and `<messageFlow>`s ONLY when** the description names genuinely
  independent parties that exchange messages (e.g. "two pools: Bank and
  Customer", "between Customer and Bank"). This mirrors the pool definition we
  just added to the editing planner: a pool = one independent party; cross-pool
  communication = message flows; sequence flows never cross a pool.
- Within each participant's process, keep the existing connectivity contract
  (every flow node lists `<incoming>`/`<outgoing>`; gateways for splits/merges).
- Message flows connect a flow node in one pool to a flow node in another
  (`sourceRef`/`targetRef` across processes). Sequence flows stay within a pool.
- Still emit **semantic-only** XML (no diagram interchange) — layout is added
  downstream.

### Part B — Pool-safe layout (`apps/web/lib/layout-collaboration.ts`, new)

A server-side function `layoutCollaboration(xml): Promise<string>` that the
generator action calls instead of `layoutProcess`. It:

1. Parses the semantic XML with `bpmn-moddle` (already a dependency).
2. **Single-process / ≤1 participant → delegate to `layoutProcess(xml)`**
   unchanged. This keeps every existing non-pool diagram byte-for-byte on the
   current path.
3. For a collaboration with N participants, lay out **each participant's
   process independently** by reusing the upstream engine: build a standalone
   `<definitions>` wrapping just that `<process>`, call `layoutProcess`, parse
   the result, and harvest its DI (every `BPMNShape` bounds + `BPMNEdge`
   waypoints). This reuses bpmn-auto-layout's node placement rather than
   reimplementing it.
4. **Assemble one `BPMNDiagram`/`BPMNPlane`** on the collaboration:
   - Stack pools vertically. For pool *i*: compute the bbox of its laid-out
     shapes; translate every shape/edge of that process by
     `(LABEL_BAND + PAD_X, cursorY + PAD_Y)` so it sits inside the pool body to
     the right of a left name-band.
   - Emit a `Participant` `BPMNShape` with `isHorizontal=true` whose bounds
     enclose the translated process + band + padding.
   - Advance `cursorY` by `poolHeight + POOL_GAP`.
   - Constants: `LABEL_BAND=30`, `PAD_X=PAD_Y=30`, `POOL_GAP=60` (tunable).
5. **Message-flow edges:** for each `<messageFlow>`, resolve source/target
   shapes (already placed), and synthesize an orthogonal vertical `BPMNEdge`:
   exit the bottom-center of the upper element, enter the top-center of the
   lower element (or nearest pool edge), with a centered label. bpmn-js renders
   message flows dashed from the `messageFlow` type — we only supply geometry.
6. Serialize the merged definitions back to XML via moddle and return it.

### Wiring

`apps/web/lib/actions.ts::generateDiagramFromPrompt` replaces
`await layoutProcess(semantic)` with `await layoutCollaboration(semantic)`.
Because `layoutCollaboration` delegates to `layoutProcess` for the
single-process case, non-pool generation is unaffected. The existing post-layout
validation (`parseBpmnXml` + `countDiEdges(...) >= flows.length`) still holds:
message-flow `BPMNEdge`s are additive, sequence-flow count is unchanged.

## Components & boundaries

| Unit | Responsibility | Depends on |
|------|----------------|-----------|
| `generate-bpmn.ts` (prompt) | Decide pools vs lanes; emit semantic collaboration | — |
| `layout-collaboration.ts` (new) | Collaboration-aware DI assembly; delegates single-process to upstream | `bpmn-moddle`, `bpmn-auto-layout` |
| `actions.ts` (wiring) | Call `layoutCollaboration`; existing validation | above |

`layout-collaboration.ts` is pure (xml → xml), no DOM, no network → unit-testable
headlessly with the real `layoutProcess`.

## Testing

- **Unit** (`layout-collaboration.test.ts`): a fixture two-pool semantic XML →
  `layoutCollaboration` → assert: `<collaboration>` preserved; each participant
  has a `BPMNShape`; every flow node has a `BPMNShape`; the two pools' y-ranges
  are disjoint (no overlap); each `<messageFlow>` has a `BPMNEdge` with ≥2
  waypoints; output reparses via `parseBpmnXml` with no errors. A single-process
  fixture → output identical to `layoutProcess` (delegation).
- **Live**: run the user's exact two-pool prompt in-app; screenshot; compare to
  `~/Desktop/twopool.png` (stacked Customer/Bank pools + vertical dashed message
  flow Submit payment → Payment received).

## Risks

- Coordinate math + moddle DI object construction is fiddly. Mitigated by
  reusing `layoutProcess` per pool (no node-placement reimplementation) and by
  the disjoint-bbox/edge unit assertions.
- Over-pooling (model emits pools for a single-party process). Mitigated by the
  conservative prompt default + the editing-planner pool definition wording.
